# Project Context

## 1. Project Overview
SwipeClean (display name "SwipeClean"; Android package `com.swipeclean.free`, slug `swipeclean-free`) is an Expo/React Native mobile app for cleaning a device media library. The product direction is a mobile-first photo/video cleanup utility: users grant media-library access, review media with a swipe-based keep/delete flow, queue items for deletion, permanently delete selected items after confirmation, inspect storage stats, and compress large photos/videos to save space.

The current app is primarily Android-oriented because the repository includes a generated native Android project, Android foreground-service permissions, and a background compression worker. iOS config exists in `app.json`, but the native iOS project is not present.

There is no custom backend, database server, or authentication provider in the current codebase. State is local to the app and persisted on-device through AsyncStorage. Subscription validation is delegated to RevenueCat's hosted service and the native app-store billing SDKs.

- State management: Zustand 5 with `persist` middleware and `@react-native-async-storage/async-storage`.
- Media access/storage: `expo-media-library` for permissions, asset reads, asset creation, and deletion.
- Image rendering/cache: `expo-image`.
- Video playback: `expo-video`.
- Compression: `react-native-compressor` for image/video compression, video thumbnails, and file-size reads.
- Background work/notifications: `react-native-background-actions` (RNBA) for the Android foreground service — a HARD SINGLETON shared by compression and the Smart Clean scan; `expo-notifications` for progress/completion/failure notifications.
- Localization (i18n): `i18next` + `react-i18next` + `expo-localization`. 10 languages (en, es, pt-BR, fr, de, it, id, hi, ar, ja); `en.json` is the canonical key source; parity enforced by `npm run i18n:check`.

Both are gated by a runtime native-capability probe (`requireOptionalNativeModule` from `expo-modules-core`) so absent native modules degrade to "not available" instead of crashing.
- Other Expo modules: `expo-haptics`, `expo-sharing`, `expo-file-system`, `expo-constants`, `expo-linking`.
- Security/app-lock: `expo-local-authentication` (Face ID/Touch ID/fingerprint) + `expo-secure-store` (Keystore-backed 4-digit passcode), both runtime-capability-probed by native-module name so they degrade safely until a native rebuild. Recurring reminder notifications use the existing `expo-notifications`.
- Gestures/animation: `react-native-gesture-handler`, React Native `Animated`, `react-native-reanimated`.
- UI/icons: React Native inline styles, `lucide-react-native`, `expo-linear-gradient`, `react-native-svg` (also used for the brand logo `AppLogo` and the adaptive launcher icon).
- Subscription/feature-gating: a Free-vs-Pro feature-access layer (`src/features/subscription/`) backed by RevenueCat (`react-native-purchases`) for Play Store / App Store / Test Store subscriptions. Pro is granted ONLY by an active subscription (`subscriptionStatus === "active"`) — the former `EXPO_PUBLIC_FORCE_PRO` build flag and the `debugProOverride` developer toggle were removed (2026-06-12).
- Native Android: generated Expo/React Native Android project with Kotlin `MainActivity`/`MainApplication`, Hermes, New Architecture enabled.
- Testing/quality gates: `tsc --noEmit` (typecheck), `node scripts/check-i18n.mjs` (locale key/placeholder parity), **Vitest** (`npm test`) unit tests for the pure modules (feature-access service, cleanup-report selectors, perceptual hash-utils), **ESLint 9** flat config (`npm run lint`, eslint-config-expo; the new React-Compiler rules are "warn", `rules-of-hooks` is an error), and a **Maestro** UI/E2E suite (`.maestro/`, run with `maestro test .maestro` against a device/emulator). Deeper review is still done by ad-hoc multi-agent passes.
- Ads: Google AdMob via `react-native-google-mobile-ads` (`src/features/ads/`) — banner + interstitial + **rewarded** (Free users watch one to unlock a single video compression), Pro-gated, TEST ads in dev / real ad units in release (rewarded uses a TEST-id placeholder until a real unit exists). A shared full-screen-ad cooldown (`full-screen-ad-gate.ts`) prevents back-to-back interstitial/rewarded ads.
- Third-party services: EAS project metadata is configured; RevenueCat is wired client-side for subscriptions and AdMob is wired for ads. Analytics, feedback, and bug reporting remain placeholders/local UI only.

## 5. Architecture and Data Flow
Expo Router owns navigation. Route files in `app/` map URLs/screens to components in `src/screens/`. Root layout mounts cross-cutting lifecycle components once above the navigation stack.

The main app state lives in `useAppStore` and is persisted to AsyncStorage under `swipeclean-free-store`. It stores permission state, selected filters, review/deletion state, compressed results, deletion history, stats, and settings. Store actions call services for native work and update state synchronously after successful results.

The gallery source of truth lives in `useMediaIndexStore` and is persisted to AsyncStorage under `swipeclean-media-index-store`. It stores indexed media by ID plus a sorted ID list, compressibility estimates, full-library savings summary, scan cursors, and scan timestamps. A quick newest-page refresh keeps recently downloaded/captured media available quickly, while the first cooperative full scan fills older months and creates a saved global cleanup estimate. Later full estimate scans are user-triggered from Cleanup.

The compression feature has a separate persisted Zustand store under `swipeclean-compression-store` (debounced storage). It stores compression jobs, media-to-job index, completed media IDs, batches, active job ID, queue, and errors. It writes through to `useAppStore.compressedMedia` after successful compression and may remove originals from the main photo list after verified deletion.

Additional domain stores (one store per domain):
- `useSubscriptionStore` (`swipeclean-subscription-store`) holds the persisted entitlement snapshot, RevenueCat billing UI state/plans, and local Test Store cancel state. The PURE `FeatureAccessService` (`feature-access.service.ts`) computes `isProUser`/`canUseFeature` from feature-flags; React reads it via `useFeatureAccess`, imperative code (e.g. the launch-time scan resume) via `getState()`. `usePaywallStore` is a tiny non-persisted open/close store; one `<ProUpgradeSheet/>` mounted in the root layout renders from it.
- `useCleanupEventsStore` (`swipeclean-cleanup-events-store`) is an append-only, newest-first ledger capped at 2000 events. Reports are derived by PURE selectors in `src/features/advanced-stats/` via `useMemo([events])` — never a store-side aggregate (avoids the fresh-object-selector crash class).
- `useSmartCleanStore` is the PERSISTED Smart Clean scan runner (`swipeclean-smart-clean-results`); `useSmartCleanFeatureCache` (`swipeclean-smart-clean-feature-cache`, debounced) caches per-asset hashes; `useSmartCleanReviewStore` drives the root-mounted review sheet.

Foreground-service coordination: RNBA is a hard singleton. Compression and the Smart Clean scan each have their own thin RNBA wrapper but only one holds the service at a time. The scan acquires it only when free, else runs plain-JS and still checkpoints; compression always has priority and a backgrounded scan yields the service (one-directional) before compression acquires it.

Data flow for media cleanup:
1. `SwipeScreen` waits for store hydration, calls `loadInitialData`, and renders permission/loading/content states.
2. `useAppStore.loadInitialData` checks permission, asks `useMediaIndexStore.refreshNewestPage` for a fast first page, then starts `useMediaIndexStore.startFullScan` only if no full scan has completed yet.
3. `PhotoLibraryService.getPhotosPage` reads media pages through `expo-media-library`; `useMediaIndexStore` maps each page into indexed assets, dedupes by ID, sorts by creation time, stores estimates, and prefetches images.
4. Screens read indexed media through `useIndexedMediaAssets` or `selectIndexedMediaAsset` and derive visible media with `filterPhotosByScope` and local `useMemo`.
5. `usePhotoLibrarySync` listens for media-library changes, app foregrounding, and a 45-second active-app interval; each refresh pulls the newest page without forcing repeated full estimate scans after one has been saved.
6. Swipe/delete actions add IDs to `reviewedPhotoIds` and `markedForDeletion`.
7. If the active swipe scope has media but all items are reviewed or marked, `SwipeScreen` renders a completed state. Start Over calls `useAppStore.restartCurrentSelection`, which resets reviewed IDs for the current month/media type only when no marked-for-deletion items remain in that same scope.
8. Permanent deletion calls `PhotoLibraryService.deletePhotos`; success updates marked queue, reviewed IDs, history, stats, and removes IDs from the media index.

Data flow for compression:
1. Cleanup screens identify compressible assets with `CompressionService.isCompressible`.
2. `createCompressionJobInput` creates queue input with estimated sizes.
3. `useCompressionStore.enqueueCompression` or `enqueueCompressionBatch` persists queued jobs and starts the runner. A batch carries an upfront `originalPolicy` (`"delete" | "keep" | "ask"`, default `"ask"`); `"delete"`/`"keep"` are auto-applied at batch completion and suppress the post-batch decision sheet.
4. `runCompressionJob` wraps compression in `CompressionNotifications.runInForeground`, updates progress, and delegates native compression to `compressMediaJob`. `verifyCompressedOutput` then runs BEFORE `saveToLibrary` and rejects an output that is not smaller than the original (`finalSizeBytes >= knownOriginalSize`), so a no-gain re-encode fails the job instead of writing a larger copy to the gallery.
5. Completed jobs save `CompressedMediaItem` into `useAppStore.compressedMedia`, update completion prompts/batches, and apply post-compression original policy.
6. Deleting originals uses `PhotoLibraryService.deletePhotos` only after output verification and policy/user decision.

Data flow for background savings estimates:
1. `useMediaIndexStore.startFullScan` reads media pages in chunks and yields briefly between pages.
2. Each page is checked with `CompressionService.isCompressible` and `CompressionService.estimate` while being stored in the index.
3. The index summary persists global scanned count, eligible count, original bytes, compressed bytes, and estimated saved bytes.
4. Cleanup renders the estimate pill from the persisted global index summary instead of reducing over the visible list during render.
5. Cleanup includes an Estimate now/Estimate again button that forces a new full scan and replaces the saved estimate.
6. Already-compressed source IDs are ignored when a scan starts; deleting originals also removes those IDs from the index.

Data flow for Smart Clean (Pro):
1. The screen (or the launch-time auto-resume) ensures a media-index full scan exists, snapshots the indexed assets, and calls `useSmartCleanStore.runScan`.
2. The runner iterates `SMART_CLEAN_SCAN_ORDER` (cheapest first) SEQUENTIALLY, inside the RNBA foreground service when free. Each detector reads/writes per-asset hashes through the persisted `featureCacheApi`, so expensive work is computed once and reused.
3. After each detector the runner checkpoints (persisted, debounced); `phase` "scanning" persists as "interrupted". On next launch `resumeIfInterrupted` (fired when both stores are ready / on app-foreground) seeds completed detectors from the persisted results and continues at the first unfinished one — provided a library `runSignature` still matches.
4. Results persist compactly (mediaIds only) and are expanded ONCE against the media index on hydrate (pruning deleted items, recomputing counts) into a stable `resultsByKey` the screen subscribes to.
5. Confirmed deletions route through `useAppStore.permanentlyDeleteMarked(ids, { emitDeletionEvent: false })`, then emit a `smartCleanActionConfirmed` event, prune the feature cache, and re-scan from the post-deletion snapshot. Deletion is blocked under non-`granted` (limited) access.

Data flow for feature gating:
1. `SubscriptionSync` in the root layout initializes RevenueCat after the subscription store hydrates and refreshes customer info when the app returns to foreground.
2. RevenueCat customer info is mapped to the local subscription snapshot from the `CleanSwipe Pro` entitlement; active entitlements unlock Pro, expired/missing entitlements fall back to Free.
3. A component calls `canUseFeature(featureKey)` from `useFeatureAccess`. Free features are always allowed; Pro features require `isProUser` (`subscriptionStatus === "active"` only).
4. On denial it calls `usePaywallStore.open(featureKey)`; the global `ProUpgradeSheet` shows a per-feature explanation and routes to the Premium tab.

Errors/loading states are mostly local and store-backed:
- Permission/loading/error state comes from `useAppStore`.
- Compression failures are normalized into friendly messages in `compression.store.ts`.
- Native service failures often return safe defaults, for example empty photo arrays or fallback estimates.
- Some errors are swallowed intentionally for prefetch, thumbnails, notifications, and external opening.

## 7. Important Business Rules
- The app must have photo/video library permission before reading or deleting media.
- App-readable permissions are `granted` or `limited`; all other statuses should show permission recovery UI.
- Initial and refresh photo loads request the newest page using a page size of 80 so recent downloads/captures can appear quickly.
- Full-library discovery/estimation runs through `useMediaIndexStore.startFullScan` in pages of 60 and yields briefly between pages to reduce UI blocking.
- The app starts a full scan automatically only when no full scan has completed yet. After a saved estimate exists, refreshes should not repeatedly re-estimate.
- The media index is the source of truth for gallery assets. `useAppStore.photos` is kept empty for compatibility and should not be used for new UI.
- `PhotoAsset.monthKey` drives month grouping and filters.
- Visible swipe media excludes already reviewed IDs and items currently marked for deletion.
- A swipe scope with existing media but zero visible cards is considered complete, not empty.
- Starting a completed swipe scope over resets reviewed IDs only for the current month/media type.
- Start Over is blocked when the current scope still has marked-for-deletion items; the user must review/delete or restore those items first. This blocked state uses a custom in-app dialog, not the native Android alert.
- Next Month moves to the next older concrete month for the active media type. If there is no next month, the UI opens the month selector.
- Left swipe means delete/mark for deletion; right swipe means keep.
- Only the active swipe card should autoplay video. Background stack cards must remain static previews.
- Swipe-card videos loop automatically and are controlled only by mute/unmute from the card; pausing is reserved for the full preview/viewer.
- Full video previews should use in-app playback with loop and explicit play/pause controls.
- Marked items are deduplicated by `photoId`.
- Permanent deletion should only remove items from state after `PhotoLibraryService.deletePhotos` succeeds.
- Demo IDs starting with `demo-` are treated as deletable without native deletion, although demo data is not currently wired into app loading.
- Compression is allowed only for photos and videos.
- Compressible videos are those with at least 20 MB estimated/known size or at least 20 seconds duration.
- Compressible photos are those with at least 5 MB estimated/known size or at least 8,000,000 pixels.
- Cleanup filtering is based on indexed media, not on scroll-loaded media.
- Cleanup estimated savings are a single persisted global estimate for all indexed eligible media. The estimate is not split per photo/video filter.
- Cleanup exposes Estimate now/Estimate again to force a new full scan and replace the saved estimate.
- Cleanup Compress All is disabled until a saved full estimate exists, so it targets the full indexed cleanup opportunity rather than only the newest page.
- Compression qualities are `low`, `medium`, and `high`; default UI quality is `medium`.
- Compressed output must have a positive verifiable size before the original can be deleted.
- If compression saves no space, the UI should prefer keeping the original or deleting the compressed copy.
- Post-compression original policy can be `ask_every_time`, `keep_original`, or `delete_original_after_success`.
- Batch compression defers original-deletion decisions until the batch finishes.
- Deleting originals cannot be undone and must remain clearly marked as destructive.

Subscription / feature gating:
- `src/features/subscription/feature-flags.ts` is the SINGLE source of truth for which features are Free vs Pro. Move a feature between tiers there; nothing else changes.
- Pro is unlocked only by an active RevenueCat `CleanSwipe Pro` entitlement mapped to `subscriptionStatus === "active"`. The old `__DEV__` debug override and `EXPO_PUBLIC_FORCE_PRO` build unlock were removed and must not be reintroduced as entitlement paths.
- Gate every Pro feature with `canUseFeature(key)` and open the paywall on denial — no ad-hoc `isPro` checks in components.
- Ads render only for non-Pro users.

- RevenueCat offering/package convention: entitlement id `CleanSwipe Pro`, offering id `default`, packages/products mapped to `monthly` and `yearly`.
- RevenueCat API keys are build-time public env vars. For development-only Test Store flows, set `EXPO_PUBLIC_REVENUECAT_USE_TEST_STORE=1` plus `EXPO_PUBLIC_REVENUECAT_TEST_API_KEY`. For real store builds, leave the Test Store flag unset and use `EXPO_PUBLIC_REVENUECAT_ANDROID_API_KEY` / `EXPO_PUBLIC_REVENUECAT_IOS_API_KEY`.
- If RevenueCat keys or offerings are missing, Premium shows a configured error and disables purchase buttons instead of granting Pro or crashing.
- There is no custom backend for subscriptions; purchases, restores, and entitlement refreshes go through RevenueCat anonymous app users.

Smart Clean (Pro):
- Detectors run cheapest-first (`SMART_CLEAN_SCAN_ORDER`); cards display in `SMART_CLEAN_DETECTORS` order and each fills when its detector completes.
- Detection NEVER fabricates results; a missing native capability degrades a detector to "not available". Per-asset hashes are cached and invalidated by the asset's modification time.
- Keeper protection is enforced in THREE places (initial selection excludes keepers, toggle rejects keepers, confirm filters keepers). Smart Clean deletes ONLY under full (`granted`) access and ONLY via the single audited `permanentlyDeleteMarked` path.
- The scan keeps running while the app is backgrounded (foreground service) but cannot execute while the process is dead — it auto-resumes from its checkpoint on next launch. Manual Stop does NOT auto-resume; an interrupted (killed) scan does.

Advanced Stats (Pro):
- Cleanup events are emitted ONLY on terminal success paths; cancelled jobs never count. "Space reclaimed" counts actual deletions, not compression-only savings.

## 10. Known Issues / Technical Debt
- Lint (`npm run lint`, ESLint 9) and unit tests (`npm test`, Vitest) were added 2026-06-13; there is still no formatting script. Test coverage is currently only the pure modules (feature-access service, cleanup-report selectors, hash-utils) — store logic, screens, and services remain untested. The React-Compiler lint rules are set to "warn" and the codebase has ~66 advisory lint warnings not yet swept (mostly `react-hooks/refs|immutability|set-state-in-effect`).
- Still placeholders (UI/state only): feedback and bug report. RevenueCat billing is implemented client-side and verified end-to-end against the RevenueCat Test Store on device (a purchase unlocks Pro via the `CleanSwipe Pro` entitlement); real Play Store / App Store purchase + Restore verification still requires real store products and a store sandbox. Google AdMob is integrated and device-verified with TEST ads (banner + interstitial); a GDPR/UMP consent gather (`AdsConsentService`, fail-open) was wired before SDK init on 2026-06-13, but a real-ads release still needs the consent form/message configured in the AdMob console, a Play Data-Safety declaration, and an iOS AdMob app. (Language selection, subscription gating, Smart Clean, Advanced Stats, Account & Security, and reminder notifications are implemented; analytics/usage/error-reporting and the Privacy section were removed from Settings - see Feature History.)
- Smart Clean cold-cache cost: the FIRST scan after a data wipe rebuilds the media index (per-asset `getAssetInfoAsync`) AND recomputes every perceptual hash; the Tier-2 pixel detectors decode every photo. On a large library (~3,000 photos / 13 GB) this is minutes and CPU-bound. The feature cache makes subsequent scans fast. A further ~2× win is possible by computing dHash + blur from a single decode (not yet done; would need threshold retuning).
- The Smart Clean foreground-service scan survives backgrounding but NOT a killed process — it auto-resumes from its checkpoint on next launch. The compression↔scan RNBA hand-off depends on RNBA flipping `isRunning()` synchronously in `stop()`; verify on-device when both run together.
- Detector thresholds want tuning on real libraries: the screenshot detector over-matches some non-screenshots (Canva/WhatsApp/downloads); `DHASH_*`/`BLUR_VARIANCE_THRESHOLD` are untuned at scale. The meme detector is a conservative metadata heuristic (no on-device ML yet).
- The launcher icon is a VECTOR recreation of the provided logo (a pasted chat image can't be written to disk as exact bytes); to ship the exact PNG, drop it in the repo and wire it as a raster.
- Compression and deletion are high-risk native flows. They need real-device testing across Android versions, permission states, limited access, large files, low storage, and interrupted foreground jobs.
- In-app video playback is wired with `expo-video`, but still needs real-device QA for local `expo-media-library` URIs, codecs, long videos, audio focus, and interaction with swipe gestures.
- PARTIALLY ADDRESSED (2026-06-13, needs a native rebuild to verify): **6K/8K video thumbnails.** Applied fix option (a) — `android:largeHeap="true"` (~512 MB heap) + raised `SAFE_MAX_DIMENSION`/`SAFE_THUMBNAIL_MAX_DIMENSION` 4096→8192, so frames up to 8K (~140 MB ARGB) now decode while a pathological >8K source still falls back to the placeholder. This change is in the committed `android/` manifest + JS but has NOT been rebuilt/verified on device — until a `:app:assembleDebug`/`assembleRelease` + reinstall, the running APK still uses the old behavior. Root cause recap: both expo-image/Glide's `VideoDecoder` and `react-native-compressor`'s `createVideoThumbnail` call the unscaled `MediaMetadataRetriever.getFrameAtTime`, so a ≥6K frame blew the old 256 MB heap (the chronic `MediaCodec.BufferInfo` OOM; see Feature History 2026-06-12). The resolution-INDEPENDENT fix (b) — a native downscaled extractor via `getScaledFrameAtTime` (API 27+) — remains deferred. Files: `src/components/media-thumbnail.tsx`, `src/services/compression-service.ts`, `android/app/src/main/AndroidManifest.xml`.
- After adding or changing native dependencies, rebuild and reinstall the native Android binary. Hot reload/Metro updates into an older APK can crash with missing native module errors such as `Cannot find native module ExpoVideo`.
- The shared media index removes the old loaded-page limitation for Swipe, Cleanup, month selection, stats, previews, and largest-media screens after the full scan has completed and persisted.
- Full media indexing is cooperative JS/native async work, not a guaranteed OS-level background task. It yields between pages to reduce UI blocking, but very large/video-heavy libraries still need real-device performance testing.
- Newly downloaded/captured media should appear after the newest-page refresh on app launch, foreground, gallery-change event, or active-app polling. If the OS does not deliver a media-library event, it may wait until the next refresh interval.

- The media index can grow large because it persists metadata and estimates for the device library. Watch AsyncStorage size/performance on very large galleries.
- Size values may be estimated when native file size metadata is unavailable, so stats and savings can be approximate.
- RESOLVED (2026-06-13): the unused `src/constants/sample-photos.ts` and the unused components (`swipe-action-buttons`, `photo-list-item`, `review-delete-button`, `activity-calendar`) were deleted. The cheap `demo-`ID guards in the services were intentionally kept as defensive no-ops.

### 2026-06-10 - Full-scan truncation fix (distinguish page-read failure from end-of-library)
- Problem: `PhotoLibraryService.getPhotosPage` caught native `MediaLibrary.getAssetsAsync` errors and returned `{ photos: [], hasNextPage: false }` — identical to a genuine end-of-library page. `useMediaIndexStore.startFullScan` therefore treated a transient mid-scan native failure as a normal final page: the scan "completed", ran `removeUnseenAfterCompleteScan` (pruning every indexed asset past the failure point) and recorded `lastFullScanCompletedAt`. One transient error silently truncated the persisted media index, so Swipe/Cleanup/Stats all shrank until the user manually tapped "Estimate again". (This is the hazard flagged in the prior entry.)
- Fix (minimal, routes into the scan's existing error path):
  - `src/services/photo-library-service.ts`: `GetPhotosPageResult` gained an optional `error?: boolean`. The `getPhotosPage` catch now returns `{ photos: [], hasNextPage: false, error: true }`, making a native read failure distinguishable from an empty/end page. Other callers (`refreshNewestPage`, `getPhotos`/`getLargestPhotos`) don't read the field, so their treat-as-empty behavior is unchanged.

A superseding scan (token changed mid-await) bails quietly via `return`.

### 2026-06-12 - Fix: chronic OOM crashes from full-res video-thumbnail decoding (+ safe video thumbnails)
- Symptom: the app hard-crashed repeatedly (10+ `data_app_native_crash` in one day per Android dropbox), typically while scrolling video-heavy grids (Largest, Smart Clean review, Compress, etc.). - Root cause: a video's first frame is decoded at FULL resolution. The S24's `largeHeap` is OFF (256 MB cap). Pre-existing (crashes predate today's UI work).
- Fix (JS-only; no rebuild, no new dependency):
  - New `src/components/video-thumb-placeholder.tsx` — `isVideoUri()` (matches local `…/name.mp4`-style URIs) + a non-decoding `VideoThumbPlaceholder` tile (play glyph on a surface).
  - `src/components/cached-image.tsx` — now renders the placeholder for any video URI UNLESS the new `allowVideo` prop is set, so a stray video URI can never trigger a full-res decode.
  - New `src/components/media-thumbnail.tsx` — the shared grid/list thumbnail. Photos pass straight through `CachedImage`. Videos render their REAL frame (via `CachedImage allowVideo`) with a play badge overlay ONLY when the source resolution is known and ≤ `SAFE_MAX_DIMENSION` (4096); otherwise the placeholder. Missing dimensions are looked up from the media index by id (`selectIndexedMediaAsset`), since e.g. `SmartCleanItem` doesn't carry width/height.
  - `src/services/compression-service.ts` — `createThumbnail` now rejects videos whose max dimension is >4096 or unknown (callers already fall back to the placeholder).   - `src/components/swipe-photo-card.tsx` — the stack-card thumbnail uses `MediaThumbnail`; the `Image.prefetch` of upcoming cards now excludes video URIs (prefetch was decoding video frames too).

A real fix for those would need `android:largeHeap="true"` (one native rebuild) or a native downscaled extractor (`getScaledFrameAtTime`); both deferred per the no-rebuild constraint.

### 2026-06-14 - Video compression made effective (bitrate-targeted) + batch lag fix
- Bug (reported + reproduced on-device): "Compress All" on videos was laggy and every video "failed". Device diagnostic (`[CompressDiag]` instrumentation, since removed) showed the real cause: `react-native-compressor`'s `compressionMethod: "auto"` re-encodes already-low-bitrate clips at a resolution-based bitrate that is HIGHER than the source, producing files 1.5–4.5× LARGER than the original (e.g. 3.6 MB → 16.5 MB, 7.2 MB → 11 MB). The no-shrink guard then correctly rejected them → a wall of "Failed", and each wasted ~11 s encode = the lag.
- Fix (in `src/services/compression-service.ts`, shared / both platforms):
  1. **Manual bitrate targeting** — `compressVideo` now uses `compressionMethod: "manual"` with `bitrate = min(profile.videoTargetBitrate, sourceBitrate * profile.videoBitrateRatio)` (caps per profile: low 1.8 Mbps/0.5, medium 3.5 Mbps/0.6, high 6 Mbps/0.7). The target is always below the source bitrate, so output is genuinely smaller. Falls back to `"auto"` only when duration/bitrate is unknown.
  2. **Bitrate-aware `isCompressible`** — a video is offered only if its source bitrate ≥ `VIDEO_MIN_COMPRESSIBLE_BITRATE` (4 Mbps). Already-compressed low-bitrate clips (downloads/social) are treated as already-optimized and not attempted, so no wasted encodes (the main batch lag) and no "Failed" wall. Unknown-duration videos fall back to the old `VIDEO_MIN_BYTES` size gate.
  3. **Bitrate-aware `estimate`** — compressed size = `originalBytes * (targetBitrate / sourceBitrate)` (duration cancels), so the global "Estimated savings" is realistic instead of a flat ratio that over-promised (the old "8.4 GB" type figure).
  4. **Progress throttle** (`compression.store.ts`) — mid-encode progress updates to the store + notification are throttled to ~4/sec (`PROGRESS_THROTTLE_MS` 250, edges always pass), so a long video doesn't re-render the compress grid on every native tick.
- The no-shrink verification guard stays as the final safety net (a re-encode that still isn't smaller is rejected before save).
- Verified on-device (S24): after the fix a 63 MB camera video compressed to 41.6 MB (**saved 21 MB**, valid) and photos to ~150 KB; the auto-method "bigger output" failures are gone. Note: the bitrate gate re-classifies the compressible pool only after an **Estimate again** (the persisted media index keeps the prior classification until then).
- Files: `src/services/compression-service.ts`, `src/features/compression/compression.store.ts`, `PROJECT_CONTEXT.md`. No new deps; JS-only (no native rebuild). `VIDEO_MIN_COMPRESSIBLE_BITRATE` and the per-profile bitrate caps/ratios are the tuning knobs. Diagnostic confirmed `ACCESS_MEDIA_LOCATION` is NOT involved (no cache-copy fallback fired).
